This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

**air** is a Claude Code plugin for automated PR code review, with an optional Managed Agent mode for CI automation.

The CLI plugin (`plugins/air/`) is markdown files and JSON metadata â€” no build system or dependencies. The Managed Agent (`managed/`) adds Python scripts for Anthropic's cloud API + a GitHub Actions workflow.

Two CLI commands: `/air:review` (13-step review pipeline) and `/air:learn` (wiki cleanup/regeneration). For CI: `managed/review.py` or the reusable GitHub Action.

## Architecture

**Review pipeline** (`commands/review.md`): Parses args, fetches PR data via the `gh` CLI, runs up to 6 agents + optional Codex in parallel (the UI/copy reviewer joins only on user-facing diffs), passes results through a verification agent that filters false positives (confidence < 60 = dropped), then posts a consolidated comment.

**Agents** (`agents/*.md`): Stateless markdown prompt files. Each is a specialized reviewer personality that receives the same rich context block (PR diff, blame data, wiki patterns, project memory). Tiered models: code-reviewer and security-auditor run on Opus 4.8 with fast-mode speed (~2Ã— faster generation; the fast-mode premium is not billed on Managed Agents sessions â€” "inference speed is managed by the runtime" per the pricing docs â€” but on the raw Messages API fast Opus 4.8 bills $10/$50 per MTok vs $5/$25 standard); review-verifier, simplify, and ui-copy-reviewer run on Sonnet 4.6; git-history-reviewer runs on Haiku 4.5. Each agent declares its model (and optional `speed:`) in frontmatter; managed resolves aliases via `MODEL_ALIASES` in `managed/setup.py`. **Current frontmatter override (the source of truth):** code-reviewer + security-auditor are temporarily pinned to Sonnet as well (`#169`, marked *temporary*), so the live fleet currently runs **Sonnet for all four review lenses + Haiku for git-history** â€” the Opus 4.8 tiering above is the intended default, to be restored when the workspace model-access constraint lifts. (Trust the `model:` line in each `agents/*.md`, not this prose, for the live tier.)

**Verification** (`agents/review-verifier.md`): Post-review quality gate. Reads actual source at flagged lines, classifies findings as CONFIRMED/DOWNGRADED/IMPROVEMENT/PRE-EXISTING/ACCEPTED PATTERN/FALSE POSITIVE using git blame decision tree.

**Pattern storage**: Two backends. **Store-backed repos** (migrated via `managed/migrate_wiki_to_store.py`) keep patterns in a per-repo Anthropic memory store â€” per-author files under `/authors/<login>.md`, shared files at the root, counter at `/meta/air-meta.json`; review sessions mount it read-only (PR content is untrusted â€” writes happen deterministically in `managed/pattern_writer.py` post-review), learn sessions mount read-write to curate it. The git wiki is an **exported mirror** rendered by a deterministic Python step (`managed/render_store_to_wiki.py`, the inverse of the migrate split), NOT by the AI session: it runs throttled after each review (â‰¤1Ã—/hour, gated by `meta.py mirror-due` â€” a cheap meta read in the common case, git push only when stale) and authoritatively after each `/air:learn` curation. Managed-only â€” the CLI never renders the store (a CLI-only store repo sees a stale wiki between managed runs). **Legacy repos** store everything on the repo's wiki (REVIEW.md, REVIEW-HISTORY.md, PROJECT-PROFILE.md, GLOSSARY.md, ACCEPTED-PATTERNS.md, SEVERITY-CALIBRATION.md, REVIEW-ARCHIVE.md) with the counter in `.air-meta.json` at the wiki root. Auto-cleanup every 15 reviews or 14 days (with â‰¥1 new PR) â€” both backends via `plugins/air/lib/meta.py` (a repo's store presence, discovered by name `air-patterns <owner>/<repo>`, IS the rollout flag). The per-review counter step is `meta.py claim` â€” an **atomic bump + learn-slot lock** (replaces the old bump+check pair): when a review crosses the threshold it acquires `learn_claimed_at` in the same CAS write, so concurrent reviews on a busy repo can't each fire a learn (the 2026-06-20 storm: a stuck counter re-triggered learn on every review). `reset` (run by `learn.py` on completion) releases the lock; a dead learn's lock ages out after `LEARN_LOCK_TTL_MIN` (self-healing).

**Pre-commit drift check** (`hooks/`): A `PreToolUse` hook on `Bash` fires before every Claude-driven `git commit`, runs either a repo-specific `.air-checks.sh` (if executable at repo root) or the plugin's built-in auto-detection (manifest version vs shields badge + `currently X.Y.Z` + `**Version:** X.Y.Z` across `CLAUDE.md`/`README.md`/`docs/**/*.md`). Non-zero exit blocks the commit. `/air:review` Step 3.5 and `/air:learn` Step 4.65 generate/augment `.air-checks.sh` from `PROJECT-PROFILE.md`. `git commit --no-verify` bypasses. See `plugins/air/README.md` for the three-level progression.

## Development Workflow

- Edit agent files (`agents/*.md`) or command files (`commands/*.md`) directly
- Reload in Claude Code with `/reload` or reconnect
- Test with `/air:review <pr-number>` on a repo with PRs
- `--dry-run` flag prints to console without posting online
- After receiving a review, fix findings and run `/air:review --respond` to auto-classify, self-check, and reply

## Key Design Decisions

- **All agents run in parallel** â€” bottleneck is the slowest agent, not the sum
- **Batched API calls** â€” Step 4 uses 3 batched `gh` calls (metadata, diff, commits), not one per field, plus a bounded sibling-PR overlap scan (â‰¤50 scanned / 10 reported)
- **Graduated dispute resistance** â€” security findings require compensating controls, style nits are readily accepted
- **Cross-repo reviews** skip local git data (blame, churn, wiki patterns) gracefully
- **Self-review mode** (`--self`) outputs a fix plan grouped by file; `--self --fix` auto-applies
- **Re-review mode** generates inter-diff from `REVIEWED_AT_SHA`, tracks FIXED/NOT FIXED per finding
- **Re-review severity-pin + deferred-findings ledger** (default ON, kill switch `AIR_LEDGER_PIN` âˆˆ `0`/`false`/`no`; both managed + CLI, single-sourced in `plugins/air/lib/verdict.py`) â€” makes severity carry-forward and finding-persistence a HARD deterministic guarantee, closing re-review's #1 failure mode: a prior `blocker` on code that *didn't change* silently drifting back to `medium` (un-gating a real blocker), or a prior finding being silently dropped. Same advisoryâ†’enforced move as `has_conflict_markers`. **Spine = finding-NUMBER identity**, with line evidence used ONLY where provably safe (the two-tier rule, validated on live fleet re-reviews): **round 3+** (prior is a re-review) pins purely by number (`INDETERMINATE`) â€” carried `#N` has no anchor and the body's only `**N.**` anchors are that round's NEW findings, whose numbers RESTART at 1 and collide with carried `#N`, so any line-evidence join cross-wires a carried blocker and could un-gate it (the bug the dogfood review caught). **Round-1â†’round-2** (the most common re-review; prior is a *fresh* review with non-colliding anchors) USES hunk-level line evidence: `extract_fresh_findings` enumerates the fresh findings by number+section-severity, `extract_fresh_finding_locations` recovers their anchors, and `finding_changed` marks a finding CHANGED iff its line falls in an edited hunk's old-side window â€” so a real fix (incl. additive/refactor) is HONORED, a fake `FIXED` on an untouched file is rewritten to `NOT FIXED`. Without this, pure-number-identity over-gated ~70% of round-2 re-reviews (every genuine fix); hunk-level dropped that to ~10% (only cross-region fixes â€” fix at a different line than the anchor â€” failed safe), and a **file-level cross-region exemption** then cleared that residual: a `FIXED` on a finding whose anchor line is UNCHANGED but whose FILE was edited elsewhere in the inter-diff (a real upstream/helper/refactor fix) is now HONORED, while a `FIXED` on a file the dev never opened (file absent) is still rewritten to `NOT FIXED` (`LedgerEntry.file_touched`; the exemption is `cross_region_fix = change==UNCHANGED and file_touched`, keyed on UNCHANGED so a stubbed/no-anchor INDETERMINATE finding stays the conservative pin). The verifier prompt (`prompts.py` / `review.md` Step 6) is aligned to match â€” a cross-region fix is a legitimate `FIXED`, judged by reading the current source, not by whether the exact flagged line is in the diff. `pin_and_resurrect` pins severity via `_max_severity` (reverts a downgrade, preserves an escalation â†’ gate only gets *stricter*, never un-gate), rewrites a fake `FIXED` on a medium+ unchanged finding to `NOT FIXED` (an explicit low/nit `FIXED` is trusted), rewrites a `DEFERRED` to `NOT FIXED` only when it's a `blocker` or its code changed (a non-blocker `DEFERRED` on unchanged code stays deferred), tolerates a `**`-bolded status, keeps `DISPUTED`/`FALSE POSITIVE`/`PRE-EXISTING` as evidence-bearing exits, and **resurrects** any prior `#N` silently absent. Managed builds the ledger in `review.py`; CLI in `review.md` Step 11.5 (`verdict.py --pin` before Step 12's `--decide`). Logs `[pin]`/`[ledger]`. The **baseline** is the NEWEST prior air review (`find_prior_review` selects by `created_at`, `id` tiebreak â€” a first-in-list walk returned the *original* review forever because the GitHub issue-comments endpoint ignores `sort`/`direction` and returns oldest-first, wedging every multi-round re-review against round 1 and false-blocking already-fixed PRs). **v1 limits (round-3+ behavior BEFORE the v2 origin-anchor below â€” read both):** round-3+ pinned by number not line (over-conservative â€” a legit late fix on a carried finding *would* stay NOT FIXED until cleared via DISPUTED/override, **now resolved by v2 origin-anchor below**; the file-level cross-region exemption applies only on the round-2 line-evidence path); the cross-region exemption is file-level, so it trusts the verifier's source-grounded `FIXED` whenever the finding's file was touched (the same trust class as a fresh-review blocker â€” the deterministic guard still independently catches downgrades, silent drops, and `FIXED`-on-untouched-file); a finding *introduced in round K* isn't pinned until it appears in a status block (per-round renumbering â‡’ cross-round anchor tracking is v2). **v2 origin-anchor (#198, default ON, kill switch `AIR_ORIGIN_ANCHOR` âˆˆ `0`/`false`/`no`; `AIR_LEDGER_PIN=0` disables it too):** round-3+ carried findings now recover their first-raise anchor by walking the air-review chain (`find_origin` over bot-authored `## Code Review` bodies, anti-spoof) and test it against the WIDER `origin..head` window instead of `baseline..head` â€” un-poisoning a fix that landed BEFORE the current baseline (the qai-be #1290 false-block: a blocker genuinely fixed in round-2 was absent from `baseline..head` â‡’ `file_touched=False` â‡’ `FIXED`â†’`NOT FIXED` forever, even though the verifier read current source and emitted FIXED). Gate-safe by a **monotone ANCESTOR gate**: `github_client.fetch_compare_status` must be `ahead`/`identical` (origin is an ancestor of head â‡’ `origin..head` is a clean SUPERSET of `baseline..head` â‡’ `file_touched` can only flip Falseâ†’True), and the flip feeds ONLY `pin_and_resurrect`'s already-validated `cross_region` trust class (0 fleet un-gates) â€” never a new trust surface; a non-ancestor (rebase/force-push) / API-fail / missing / disabled origin falls back to v1 number-identity (conservative, never un-gates). The verifier reads current source, so a fake FIXED still can't pass. The **reconstructive** walk needs ZERO change to the frozen 3-site status-line contract â€” the embedded-`|origin:sha` tag was REJECTED for re-introducing that silent-drift footgun. `make_origin_resolver` single-sourced in `review.py` (managed + headless import it); the **CLI** is wired too via `verdict.make_file_origin_resolver` + `verdict.py --pin --origin-chain/--origin-diffs` (review.md Step 11.5 runs the ancestor gate LOCALLY â€” `git merge-base --is-ancestor` â€” and writes only confirmed `origin..head` diffs, keeping verdict.py network-free; a diff present âŸº ancestor-confirmed, same proof). Promote-sibling skips it (its origin isn't an ancestor of this head). Real #1290 replay confirms the un-poison (v1 gates, v2 APPROVEs); 17 new tests (incl. a chain-order regression guard: the chain MUST be oldest-first so `find_origin` recovers the first-raise anchor, not a later round's renumbered `#N`; plus the CLI file-backed resolver). **v3 cross-FILE fix + rewrite marker (qai-be #1422, round-2):** the round-2 `file_touched` signal now considers EVERY file a finding REFERENCES â€” its blob-link anchor PLUS backtick'd path tokens in its prose (`extract_finding_files` + `_referenced_file_touched`) â€” not just the single anchor, so a blocker flagged at the symptom/read site (`config/services.php`) but FIXED in the wiring file (`scripts/after_install.sh`) is HONORED. Before v3 that false-blocked a promote: the verifier emitted `FIXED` (header said "all blockers resolved") but the pin keyed `file_touched` on the anchor file only (never touched) â†’ rewrote `#1`â†’`NOT FIXED` â†’ `CHANGES_REQUESTED`, the label silently contradicting its own rationale. Still keys on a real `@@` hunk (metadata-only excluded) and requires a path-shaped token with a `/` (bare basenames excluded â€” too loose to match safely); over-crediting is the SAFE direction (it can only honor a verifier's source-grounded `FIXED`, never un-gate a dropped/downgraded finding â€” those guards are independent). And whenever the pin DOES rewrite a `FIXED`â†’`NOT FIXED`, `pin_and_resurrect` now appends `_PIN_REWRITE_MARKER` to the status line so the label is self-explaining rather than contradicting its retained rationale + the summary header (the marker is inert to the gate â€” only the STATUS token is parsed). round-3+ cross-FILE (origin path) is a possible follow-up; v3 is round-2 only.
- **Fresh-gate exposure floor** (default ON, kill switch `AIR_CATEGORY_FLOOR` âˆˆ `0`/`false`/`no`; floor *application* single-sourced in `plugins/air/lib/verdict.py`, parity across managed + CLI `--decide`) â€” the deterministic "active exposure = blocker" enforcement (same advisoryâ†’enforced move as `has_conflict_markers` and the re-review ledger). The re-review ledger pins severity by *carrying a prior finding forward*; a **fresh** review has no prior to pin against, so a weaker tier finding a real exposure but rating it `medium` would silently APPROVE (the 2026-06-12 Sonnet bench: org-wide PII + an RBAC bypass, both found, both rated medium, both would have un-gated). The verifier tags each confirmed, genuinely-exploitable exposure `[sec:<token>]` (one of a fixed blocker-class vocabulary â€” `pii-exposure`/`authz-bypass`/`sqli`/`idor`/`leaked-credential`/â€¦, the `_BLOCKER_CATEGORIES` frozenset); `count_category_floored` floors any such tag (outside the already-counted Blockers section) to a blocker for the gate, regardless of the model's own label. Inert on tag-less bodies â†’ disabling (or any run before tags exist) is byte-identical to the pre-floor gate; it can only ever make the gate *stricter* on a real exposure, never un-gate. **Single emission source (all paths):** the tag-emission rule lives in the verifier SYSTEM PROMPT (`plugins/air/agents/review-verifier.md`) â€” the system prompt for the managed verifier agent, the CLI verifier agent, AND assembled into solo by `solo_prompt.py` â€” so managed CI, the CLI, and solo all emit `[sec:<token>]` from one source (no `prompts.py` task injection). Vocabulary is locked to the frozenset by `.air-checks.sh` Check F (a Python step asserts every blocker-class token appears backtick-quoted in `review-verifier.md`).

## Releases (automated via release-please)

Never tag or cut a release manually. Every commit on main that uses Conventional Commits (`feat:`, `fix:`, `docs:`, `chore:`, `refactor:`, etc.) feeds the release PR:

- `.github/workflows/release-please.yml` runs on every push to main
- It maintains a long-lived "chore(main): release vX.Y.Z" PR on the repo
- That PR shows: next version (computed from commit types â€” `feat` â†’ minor bump, `fix` â†’ patch, `BREAKING CHANGE:` in footer â†’ major), all version-mirror files bumped atomically, and a CHANGELOG.md entry

Files tracked for version mirroring are defined in `.release-please-config.json`'s `extra-files` array. Don't maintain a second list here â€” keep the config as the single source.

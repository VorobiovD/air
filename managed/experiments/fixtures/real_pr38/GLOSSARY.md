# Project Glossary — Domain Terms

Last updated: 2026-04-25

| Term | Definition | Context |
|---|---|---|
| air | Automated code review plugin for Claude Code. Name stands for "AI Review." | project name |
| agent | A specialized markdown prompt file that acts as a review personality. Each agent receives the same PR context and produces findings independently. | `plugins/air/agents/*.md` |
| specialist | Synonym for an individual reviewer in the cloud orchestrator. The 4 specialists are code-reviewer, simplify, security-auditor, git-history-reviewer; the verifier runs after them. Codex (when enabled) is the 5th specialist. | `managed/review.py` |
| inter-diff | The diff between the previously reviewed SHA and the current HEAD. Used in re-review mode to show only what changed since the last review. Generated with `git diff <old-sha>..<new-sha>` locally or `GET /repos/{owner}/{repo}/compare/{base}...{head}` (three-dot) for managed. | `commands/review.md` Step 6, `managed/review.py` `fetch_inter_diff` |
| skip-if-unchanged | Optimization: when re-review's `prior_sha == head_sha`, exit without reviewing. Saves the ~$2 review when nothing actually changed. Implemented in `managed/review.py`; the spoof-resistance fix in PR #28 added bot-author filtering on the marker comment. | `managed/review.py` |
| wiki drift | When the actual project state diverges from what PROJECT-PROFILE.md or GLOSSARY.md describes. Agents flag drift during review; the orchestrator records it for the next `/air:learn` run. | `commands/review.md` Step 7 |
| graduated resistance | A dispute resolution mechanism with three levels (HIGH/MEDIUM/LOW) that determines how easily a developer can override a review finding. Security findings require compensating controls; style nits are accepted readily. | `commands/review.md` Step 13 |
| verification agent / verifier | The `review-verifier` agent that runs after all reviewers complete. It reads actual source code at flagged lines, assigns confidence scores (0-100), and classifies findings into 6 verdicts. | `plugins/air/agents/review-verifier.md` |
| confidence score | A 0-100 score assigned by the verifier to each finding. Below 60 = dropped as false positive. Above 60 = kept. Per-agent thresholds in SEVERITY-CALIBRATION.md can override the default 60. | `plugins/air/agents/review-verifier.md` |
| verdict | One of 6 classifications assigned by the verifier: CONFIRMED, DOWNGRADED, IMPROVEMENT, PRE-EXISTING, ACCEPTED PATTERN, FALSE POSITIVE. | `plugins/air/agents/review-verifier.md` |
| KAIROS | The codename for the REVIEW-HISTORY.md generation process in the learn command. Fetches review comments from recent merged PRs and produces finding frequency tables, file hot spots, and author trends. | `commands/learn.md` Step 4 |
| KAIROS cache | Persistent cache at `$HOME/.cache/air/kairos/` for KAIROS PR-comment fetches. Step 0 of `learn.md` deletes entries older than 30 days. Per-repo scoping is by hash-of-repo-name in the cache key. | `commands/learn.md` Step 0 |
| self-review | A mode (`--self`) that reviews local uncommitted changes instead of a PR. Produces a fix plan with exact current/replacement code per finding. No GitHub posting. Wiki pattern updates still push. | `plugins/air/commands/review-self.md` |
| re-review | A mode (`--re-review`, also auto-detected from prior comment) that generates an inter-diff from the last reviewed SHA and tracks each previous finding as FIXED/NOT FIXED/DISPUTED. | `commands/review.md` Step 6 |
| respond mode | A mode (`--respond`) that automates the developer side of the review cycle. Reads the existing review, classifies each finding as fixed/disputed/acknowledged based on local commits, runs a self-check on the fix diff, and posts a structured response. | `plugins/air/commands/review-respond.md` |
| fix plan | The output of self-review mode: a list of findings with exact current code and replacement code, grouped by file, ordered by severity. | `plugins/air/commands/review-self.md` |
| smart default | Auto-detection when no flags are passed: checks for an open PR, existing review with new commits (auto re-review), or local changes (auto self-review). | `commands/review.md` Step 2 |
| cross-repo | Reviewing a PR from a different repository than the current working directory. Skips wiki, blame, churn, and learn operations. Detected by comparing URL repo with `CURRENT_REPO`. | `commands/review.md` Step 1 |
| fresh mode | The `--fresh` flag — full review from scratch, post a NEW comment regardless of existing reviews. Bypasses the auto-detect re-review path. | `commands/review.md` Step 1 |
| full mode | The `--full` flag — review the ENTIRE codebase (diff from empty tree to HEAD). Output to console only, never posts. Used for first-time audits, small projects, or full security passes. Read-only, so `--fix` is ignored. | `commands/review.md` Step 1 |
| closed PR mode | The `--closed` flag — opt-in review of closed/merged PRs (post-merge audit, wiki backfill, dogfooding without opening a new PR). Default refuses to spend tokens on PRs nobody is looking at. Step 12 skips the approve / request-changes verdict on CLOSED/MERGED state because GitHub 422s those. | `commands/review.md` Step 5, Step 12 |
| Managed Agent | The cloud-side distribution of air. `managed/review.py` is a client-side orchestrator that creates/finds 5 Anthropic API agents, fans out 4 specialists in parallel via `asyncio.gather`, runs the verifier, and posts the comment. Deployed as the reusable `managed-review.yml` GitHub Action. | `managed/`, `.github/workflows/managed-review.yml` |
| client-side orchestrator | The architecture introduced in v1.7.0 (PR #23): the orchestrator runs as Python on the GitHub Actions runner instead of a dedicated Anthropic agent. Eliminates one billed agent per review and centralizes fan-out / cleanup logic. | `managed/review.py` |
| AIR_BOT_TOKEN | The classic GitHub PAT used by the Managed Agent to clone wiki, fetch PR data, and post review comments. Must be a classic PAT — fine-grained PATs do not support wiki push. Stored as a repo/org secret. | `managed/README.md` Setup |
| AIR_PLUGIN_ROOT | The CLI plugin's installation directory. Derived in Step 0 of `review.md` / `review-self.md` / `learn.md` from `~/.claude/plugins/cache/air/air/*/` (sorted by version). Used by the `meta.py` and drift-hook subprocesses. Cleared with a warning if unresolvable. | `commands/review.md` Step 0 |
| AIR_TMP | The session-scoped temp directory: `mktemp -d "/tmp/air-XXXXXX"`. Every `/tmp/` write within a single command invocation goes here so parallel sessions don't collide. Stale dirs (>1 day) are GC'd by the next run's Step 0. Introduced in v1.5.1. | `commands/review.md` Step 0 |
| .air-meta.json | The wiki-backed counter file (introduced in v1.8.0, PR #39) tracking `reviews_since` and `last_processed_pr`. Lives at the wiki root. Both CLI and Managed paths read/write through `lib/meta.py`. Replaces the old per-developer `~/.claude/review-learn-meta.json`. | `plugins/air/lib/meta.py` |
| wiki-backed counter | The auto-trigger counter for `/air:learn`. Now shared across the team via `.air-meta.json` in the wiki (rather than per-developer locally). Threshold: 5 reviews or 2 days, whichever fires first. The counter is committed back to the wiki on every increment. | `plugins/air/lib/meta.py`, `wiki_git.py` |
| auto-trigger cleanup | The counter-based mechanism that runs `/air:learn` every 5 reviews or 2 days. Tracked in `.air-meta.json` (wiki-backed since v1.8.0). Distributed across team members. | `commands/review.md` Step 13, `commands/learn.md` Step 7 |
| `meta.py bump` / `check` / `reset` | The three subcommands of `lib/meta.py`. `bump` increments `reviews_since` and updates `last_processed_pr`; `check` decides whether to trigger learn (returns reason); `reset` zeros `reviews_since` and bumps `last_check`. All three commit the change to the wiki via `wiki_git.commit_meta`. | `plugins/air/lib/meta.py` |
| `wiki_git.commit_meta` | Helper that stages `.air-meta.json`, commits, pushes — with `pull --rebase` retry on non-fast-forward. Used by `meta.py` so concurrent reviews can both bump without losing the increment (within the limits documented in the docstring). | `plugins/air/lib/wiki_git.py` |
| `_redact()` | Helper in `wiki_git.py` that strips `x-access-token:<token>@` from URLs in error logs. Every subprocess error path that may surface the wiki URL must route through it; the full `CalledProcessError` `__str__` echoes the `cmd` list which includes the token-bearing URL. | `plugins/air/lib/wiki_git.py` |
| drift hook / pre-commit drift | The PreToolUse hook on `Bash` (introduced in v1.6.0) that intercepts `git commit` and runs zero-config built-in checks (manifest-version vs README badge, doc-mirror grep) plus an opt-in repo-checked-in `.air-checks.sh`. Blocks the commit if drift is detected; `--no-verify` (as a flag, not a substring) skips. | `plugins/air/hooks/pre-commit-drift.py` |
| .air-checks.sh | The opt-in per-repo drift-check script consumed by the pre-commit hook. Must be executable and explicitly trusted by the developer. Generated by `learn.md` Step 3.5 when the project profile shows new manifest files. | `.air-checks.sh` (root), `plugins/air/hooks/pre-commit-drift.py` |
| built-in checks | The zero-config drift checks shipped with the hook: manifest version vs README badge mirror, agent-count mirror, structure-tree marker. Live in `plugins/air/hooks/builtin-checks.sh`. Run before `.air-checks.sh`. | `plugins/air/hooks/builtin-checks.sh` |
| release-please | The Google-maintained release automation that opens release PRs on every push to main, bumps `plugin.json` + extra-files, generates CHANGELOG entries, and creates a GitHub Release on merge. Added in PR #36. | `.release-please-config.json`, `.github/workflows/release-please.yml` |
| extra-files | The `release-please` config list of files outside `plugin.json` that carry the version marker (`<!-- x-release-please-version -->`). Files added later that carry the version are silent drift unless added here. | `.release-please-config.json` |
| platform detection | Auto-detection of GitHub vs GitLab from the git remote URL. Sets the CLI tool (`gh` or `glab`) and domain. Can be overridden with `AIR_PLATFORM` env var for self-hosted GitLab on non-standard domains. | `commands/review.md` Platform Detection |
| PR Context block | A structured data block passed to every agent containing PR metadata, CI status, blame summaries, churn data, previous PR comments, project memory, session context, and the `Wiki files directory:` field that points at the orchestrator's session temp directory plus the list of available wiki files. | `commands/review.md` Step 7 |
| `prior_review_body` | The full body of the most recent `## Code Review` comment on the PR, fetched in re-review mode and interpolated into specialist prompts as authoritative-prior-finding context. Must be `html.escape`-d and length-bounded before interpolation. | `managed/review.py` `find_prior_review` |
| own-PR guard | Logic that skips the `gh pr review` verdict (approve/request-changes) when the reviewer is also the PR author, since GitHub disallows self-approval and self-requesting-changes. | `commands/review.md` Step 12 |
| comment rot | Outdated code comments that no longer match the code they describe. The code-reviewer agent checks for function signatures that changed without docstring updates and similar drift. Closely related to "stale rationale comment" findings: rationale paragraphs that explained the old behavior survive when the surrounding code is replaced. | `plugins/air/agents/code-reviewer.md` |
| high-churn | A file with 5+ commits in the last 6 months. Flagged by `git-history-reviewer` as a potential design smell. | `plugins/air/agents/git-history-reviewer.md` |
| blame decision tree | The verifier's structured process for determining if a finding was introduced by the current PR or is pre-existing. Steps: check `+`/`-` lines in diff, then context lines via `git blame`, then file-level PR scope. | `plugins/air/agents/review-verifier.md` |
| batched API calls | Step 4 optimization: one `gh pr view` call fetches all PR metadata (title, author, files, CI status, commits) instead of separate calls per field. Total: 3 API calls for all PR data. | `commands/review.md` Step 4 |
| `_github_paginate` | The single-source pagination helper across `find_prior_review` and `fetch_comments_since`. Extracts `Link: rel="next"` once instead of duplicating it. Mirrors `api.py::_paginate` for the Anthropic side. | `managed/review.py` |
| `LIVE_SESSIONS` | The set of in-progress Anthropic API session IDs tracked by `managed/review.py`. Signal handlers and atexit cleanup interrupt every session in this set on SIGINT/SIGTERM/SIGHUP/normal exit, so a runner shutdown doesn't orphan billing. | `managed/review.py` |
| user.interrupt | The Anthropic API event payload (`{"type": "user.interrupt"}`) sent to abort an in-progress session. Issued by the cleanup helpers in `managed/review.py`. | `managed/review.py` |
| Codex | Optional GPT-5.4 reviewer from the openai-codex plugin. Provides an independent second opinion from a different model family. CLI: `codex-companion.mjs`. Managed: `npm install -g @openai/codex` + `codex review --base <sha>`. Gated on `OPENAI_API_KEY`. Skipped cleanly if unset (since PR #38 the orchestrator counts the skip as a non-degraded outcome). | `commands/review.md` Step 7, `managed/review.py` |
| Opus | The Claude model for `code-reviewer`, `security-auditor`, and `review-verifier`. Specified in agent frontmatter as `model: opus`. | `plugins/air/agents/{code-reviewer,security-auditor,review-verifier}.md` |
| Sonnet | The Claude model for `simplify` and `git-history-reviewer` since the v1.5.0 model-tiering change (PR #20). Cheaper per-token; covers the lower-stakes specialists. Cost claims must specify the per-model boundary because prompt cache is per-model. | `plugins/air/agents/{simplify,git-history-reviewer}.md` |
| model tiering | The v1.5.0 design (PR #20) that splits the 5 agents across Opus and Sonnet to reduce per-review cost. Model is parsed from agent frontmatter (`parse_agent_model`); missing/unknown models pass through to the API as-is. | `managed/setup.py` |
| pre-existing | A finding classification for issues that exist in the codebase but were not introduced by the current PR. Reported in a separate section; do not block merge. | `plugins/air/agents/review-verifier.md` |
| author pattern lifecycle | State machine governing author behavioral patterns in REVIEW.md: create (first occurrence) → strengthen (counter increments, clean-PR counter resets) → decline (5 consecutive clean PRs) → archive (10 consecutive clean PRs). Patterns are never deleted. | `commands/review.md` Step 13, `commands/learn.md` |
| dispute rate | The ratio of disputed findings to total findings for a given agent+category combination. Used to compute per-agent confidence thresholds in SEVERITY-CALIBRATION.md when 10+ data points exist. | `commands/learn.md` Step 4.5 |
| resource exhaustion | Security checklist category (items 29-31) covering event listener leaks, connection pool exhaustion, unbounded data growth. In the managed orchestrator: orphan sessions on shutdown, hung streams without `wait_for`, atexit cleanup running blocking HTTP on the loop thread. | `plugins/air/agents/security-auditor.md`, `managed/review.py` |
| REVIEW.md | Wiki page storing curated review patterns: author tendencies, service gotchas, common findings. Updated incrementally after each review. | wiki storage |
| REVIEW-HISTORY.md | Wiki page with auto-generated analytical data from PR comment history. Finding frequency, file hot spots, author trends, timeline. Regenerated by KAIROS. | wiki storage |
| PROJECT-PROFILE.md | Wiki page documenting project characteristics: languages, frameworks, services, review focus rules, applicable security checks. Generated on first run by an Opus deep scan; refreshed on `--refresh-profile` or when manifest files change. | wiki storage |
| GLOSSARY.md | Wiki page defining domain terms (this file). Refreshed each `/air:learn` run with new terms found in REVIEW.md, ACCEPTED-PATTERNS.md, CLAUDE.md, README.md. | wiki storage |
| ACCEPTED-PATTERNS.md | Wiki page storing team-approved patterns that suppress matching findings in future reviews. Populated when developers successfully dispute findings through graduated resistance. | wiki storage |
| SEVERITY-CALIBRATION.md | Wiki page with per-agent, per-category confidence thresholds computed from dispute rates. Overrides the default 60 threshold when sufficient data exists (10+ data points). | `commands/learn.md` Step 4.5 |

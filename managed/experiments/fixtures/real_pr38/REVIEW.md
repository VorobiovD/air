# Review Patterns — air

## Common Findings

- **Documentation and metadata sync**: When adding flags, changing agents, or restructuring code, update all references in lockstep — YAML argument-hints, README usage sections, CLAUDE.md descriptions, line counts, step numbers, flag lists, and badges. Mirror parity between root `README.md` and `plugins/air/README.md` is enforced by the drift hook but only for badges; "What's New" sections, architecture diagrams, agent counts, and cost figures still need manual sync. Highest-blast-radius targets: model/version bumps, agent-count changes, new commands, new flags. release-please's `extra-files` list must enumerate every file carrying the version marker — files added since the last edit are silent drift.
- **Symmetric branches in CLI orchestration**: `review.md`, `review-self.md`, and `review-respond.md` share gate logic (auto-trigger threshold, `AIR_PLUGIN_ROOT` derivation, wiki push chain). When one branch grows a new sub-step or directive, the others must mirror it. Bash-comment-only enforcement in one branch where the sibling has an explicit `>>> AUTO-TRIGGER DECISION (do NOT skip this block) <<<` directive is a flow-routing gap.
- **Unguarded flag combinations and guard placement**: New flags routed through existing flows must reject incompatible combos and run the rejection BEFORE the flow-diverters — guards placed after `--full`/`--self` short-circuits silently no-op. Recurring combos: `--closed` + `--self`/`--full`/`--respond`; `--respond` + `--fresh`/`--rewrite`/`--re-review`; `--full` + `--fix`. The same rule generalises to any cross-path directive: instructions and rejections that apply across multiple conditional paths must be placed BEFORE the first conditional, not buried inside one path with "applies to all" prose.
- **Platform-neutral wording**: After GitLab support, avoid "GitHub" in instructions that apply to both platforms. Use "PR/MR", "posts online" not "posts to GitHub". Field names: `author.login` (GitHub) / `author.username` (GitLab); `nameWithOwner` / `path_with_namespace`; `headRefOid` / `sha`. Translation table in `commands/platform-gitlab.md`.
- **Orchestrator must not fix code during review**: The review pipeline (Steps 7-12) reports findings, it does not repair them. Fixing code and posting "0 findings" defeats the review cycle. `--respond` is the only mode that touches the working tree (and only for verifying claimed fixes).
- **Two-phase API loops must avoid redundant fetches**: When Phase 1 identifies a subset and Phase 2 fetches details for that subset, Phase 2 must not re-call the same endpoint Phase 1 already paginated. `find_prior_review` and `fetch_comments_since` paginate the identical issue-comments endpoint — extract a single `_github_paginate` helper.
- **Parameter half-plumbed through both branches of a function**: When adding a new field/parameter to a function with separate create/update paths (or any if/else symmetry), wire it through BOTH branches. Silently dropping it in one branch makes the feature a no-op for that path — especially dangerous when the update path is the common case (resync of existing resources). See PR #20 `managed/setup.py` `create_or_update_agent` model-tiering drop.
- **Single source of truth for pinned strings**: API model IDs (`claude-opus-4-7`), version strings, marker prefixes (`## Code Review`), filenames (`.air-meta.json`), and pricing/cost tables duplicated across files drift on the next bump. Extract to a named constant or alias map and import from a single module. PR #20 had 4 sites for `claude-opus-4-7`; PR #28 had 3 sites for the `## Code Review` marker; PR #39 duplicated `META_FILENAME = ".air-meta.json"` between `meta.py` and `wiki_git.py`.
- **Prompt-cache and cost claims must respect per-model boundaries**: Claude's prompt cache is per-model. Cross-family cache hits across tiered agents aren't possible. When tiering across Opus/Sonnet, "N of M calls share a cached prefix" must say "within each model family" — and the M number itself must reflect the post-tiering fan-out (currently 4 specialists + verifier; 5 with Codex enabled).
- **Untrusted input must be escaped AND bounded before reaching specialist prompts**: Every user-influenced string interpolated into XML-tagged context blocks (PR title, body, commit messages, developer comments, **prior review body in re-review mode**, **inter-diff**) must be `html.escape`-d AND truncated to a bounded length. Truncate before escape — escaping a string can push it past the budget. The verifier and specialists treat `<prior-review>` content as authoritative; an unsanitized PR body containing `</prior-review>` closes the tag and exposes the remainder as untagged prompt text (PR #28).
- **PR-comment control plane must be bot-identity-scoped**: Re-review mode treats comments matching `## Code Review` as authoritative prior reviews. Without an author filter (`user.login`, `performed_via_github_app`), any PR participant can suppress reviews (post a fake "prior review" with the current head SHA → skip-if-unchanged exits) or steer re-review classifications (pre-fill "FIXED" status). Marker regex must be anchored (`^## Code Review( \(Re-review\))?\s*$`); SHA comparison must be full-string against a 40-char SHA, not prefix-match.
- **API-failure surfacing distinguishes "no data" from "fetch failed"**: GitHub helpers (`find_prior_review`, `fetch_comments_since`, `fetch_inter_diff`) collapsing a 502 / 5xx / rate-limit / 404 into the same return as "nothing found" silently mode-shifts the orchestrator. A rate-limited prior-review lookup falls through to a full $2 review; a 404 on the compare endpoint after a force-push silently exits with no review posted. Distinct sentinels per failure mode; log every non-ok response. Same shape applies to fallback resolution: `AIR_PLUGIN_ROOT` resolving to a non-existent `…/air/latest` path silently no-ops the auto-trigger (PR #39).
- **Token-bearing URLs leak through subprocess error paths**: Wiki clone URLs embed `x-access-token:<token>@github.com`. Any `subprocess.CalledProcessError` whose `cmd` list or `e.stderr` reaches stderr/logs leaks the token. CI's secret redactor masks registered secrets in Action logs but the local `claude /air:learn` terminal does not. Route every error message — including the full `str(e)` fallback when `stderr` is empty — through `_redact()` from `lib/wiki_git.py`.
- **Async lifecycle: signal/atexit cleanup, per-session timeouts, gather error semantics**: `asyncio.gather(*tasks)` cancels siblings on the first exception unless `return_exceptions=True` (orphans billing on still-running cloud sessions). Every `run_session` needs an outer `asyncio.wait_for`. Signal handlers must track `LIVE_SESSIONS` and interrupt them on SIGINT/SIGTERM/SIGHUP without doing blocking HTTP on the main thread while the loop is alive. `return_exceptions=True` outside of `gather` is a no-op.
- **CI workflow hygiene**: Pin third-party Actions to a SHA (or at minimum a major version) — floating `@v4` tags can be retagged. Every workflow needs an explicit top-level `permissions:` block (least-privilege; `contents: read` for test-only jobs). `actions/checkout@v4` defaults to `persist-credentials: true` — set `false` for jobs that don't push. Release/version-bump workflows need a `concurrency:` group. `workflow_dispatch` boolean inputs arrive as the literal strings `"true"`/`"false"` — typed `workflow_call` callers must pass strings (PR #31). Detached `Popen` from a CI job (e.g., `subprocess.Popen(learn.py, start_new_session=True)`) does not survive job teardown — run synchronously or dispatch a separate workflow (PR #39).

## Service-Specific Patterns

### `plugins/air/lib/` (Python helpers — stdlib-only)

- **Stdlib-only invariant**: Anything that needs `requests`, `anthropic`, etc. belongs in `managed/`. The CLI invokes `python3 lib/meta.py …` directly — no install step.
- **`python3 -c` heredocs must pass paths via env, not interpolate**: The `learn.md` Step 7 pattern of interpolating `$WIKI_DIR` / `$AIR_PLUGIN_ROOT` directly into a Python string literal is a path-with-quote / path-with-backslash injection. Use `VAR=$VAR python3 -c "import os; … os.environ['VAR']"`.
- **`commit_meta` retry semantics**: `wiki_git.commit_meta` uses `pull --rebase` retry on non-fast-forward. The retry handles concurrent counter bumps but conflict-on-rebase drops the local change — document the behaviour honestly in the docstring (no claims of commutativity that don't hold; PR #39).
- **Pytest coverage for new helpers**: Each new public function in `meta.py` / `wiki_git.py` ships with at least one happy-path test and one boundary/error test. The wiki-counter race is not yet covered (PR #39); add a test that simulates two concurrent bumps.

### `managed/` (Python orchestrator)

- **Async lifecycle (gather, wait_for, signal/atexit)**: `asyncio.gather(*tasks, return_exceptions=True)` — all 4 specialists in one batch (5 with Codex when `OPENAI_API_KEY` is set); one specialist crash must not cancel the others. Outer `asyncio.wait_for` on every `run_session` (and on the Codex `npx` subprocess — single layer, not nested; PR #38). Live sessions tracked in `LIVE_SESSIONS` and interrupted on SIGINT/SIGTERM/SIGHUP. Module-import-time `signal.signal(...)` and `atexit.register(...)` are fine, but the cleanup function must not call blocking HTTP on the main thread while the asyncio loop is alive (PR #27); use `loop.add_signal_handler` where possible.
- **Don't `Popen` `learn.py` from a CI job**: `start_new_session=True` detaches from the process group but not from the runner VM. The job tears down before the detached child finishes. Either run synchronously (`subprocess.run([... , "--poll"])`) or dispatch a separate workflow (PR #39).
- **Don't pass full `os.environ` to subprocesses**: `setup.py` only needs the API keys it uses. Passing `os.environ` leaks unrelated secrets into the child's `/proc/<pid>/environ`.
- **GitHub error bodies bounded in logs**: `:200` / `:500` slices on response bodies in error paths (`managed/review.py:94, 335`). Don't `print(resp.text)` raw. Apply the same shape to `resp.json()` callers (`generate_app_token`, `test-parallel.py` environment fetch) — check status before parsing.

### `plugins/air/hooks/` (Pre-commit drift)

- **Auto-execution of repo-checked-in scripts is a supply-chain escalation**: `.air-checks.sh` shipping in any cloned repo would auto-run on the first commit. Gate behind executable-bit check + repo-trust prompt + clear opt-in.
- **`--no-verify` substring match bypasses on commit-message text**: A commit with body `"--no-verify in some context"` would trigger the bypass. Match `--no-verify` only as a flag (regex with word boundary or argv parse), not as a substring of the full command.
- **Cross-shell portability**: Glob patterns like `docs/**/*.md` silently fail on macOS bash 3.2 (no `globstar`). Use `find` for recursion.
- **Bypass coverage**: Drift detection must catch `git -C <path> commit` and `git --git-dir=X commit`, not just bare `git commit`.

### `.github/workflows/` (CI)

- **Reusable workflow input typing**: `workflow_dispatch` inputs come through as strings; reusable callers passing them to typed `workflow_call` inputs must declare strings, not bools. `default: 'true'` (string), not `default: true` (bool).
- **release-please `extra-files`**: Every file carrying the version marker (`<!-- x-release-please-version -->`, badges, prose) must be enumerated. Markers inside fenced code blocks render literally — extract them out of the fence (PR #36).
- **Pin third-party actions**: `googleapis/release-please-action@v4` is a floating tag and should be pinned to a SHA or major-version pin reviewed in releases. `npm install -g @openai/codex` is unpinned and is a fresh supply-chain surface (PR #38).
- **Concurrency on release workflows**: release-please needs a `concurrency:` group to prevent parallel runs from racing.

## Author Patterns

### VorobiovD

- **Aspirational comments and inaccurate-claim prose** (8x: #11, #20, #21, #23, #27, #28, #28, #39 | last 0 PRs: 0 clean): Describes capability without implementing it, or with optimistic arithmetic, or carries claims that don't match the implementation in the same commit. Recurring forms: cost/cache claims that assume pre-tiering fan-out; docstring promises (`"functionally equivalent to two-dot"`, `"counter semantics are commutative"`) that hold only in the easy case; comments that overstate guarantees ("prevents the exact leak we're trying to prevent" when it only narrows the race window); TODO-style "would be nice" inline comments left in shipped code. Tendency: if the prose describes a quantitative outcome of a behavior change in the same commit, re-derive the number after the change lands. If the docstring says "equivalent" or "atomic" or "commutative", state which case it covers and which it doesn't.

- **Flow routing gaps and symmetric-branch drift** (15x: #1, #8, #11, #12, #20, #21, #22, #23, #28, #30, #33, #38, #39, #39, #39 | last 0 PRs: 0 clean): Adds features that route through existing flows but misses updating all guards, branches, or symmetric helpers. Recurring forms: parameter wired through `create` path but not `update` path (PR #20 `create_or_update_agent`); `--closed` guard placed after `--full`/`--self`/`--respond` flow-diverters where it can't be reached (PR #33); `review.md` getting an explicit `>>> AUTO-TRIGGER DECISION <<<` block while `review-self.md`'s symmetric branch enforces the same skip via a bash comment (PR #39 #2); new `$AIR_PLUGIN_ROOT` invocation added to `learn.md` Step 7 without the matching derivation in `learn.md` Step 0 (PR #39 #3); GitHub Actions still passing a `--poll` flag the orchestrator no longer accepts (PR #23 #1); CLI `/air:learn` still resetting the old local counter while managed resets the new wiki-backed one (PR #39 #1). Tendency: when adding a parameter, flag, or env var to one site, audit every code path in that function/file/sibling-command before committing — and if the change makes a flag combination newly meaningful, add the rejection guard before the flow-diverters.

- **Fragile inline enumerations and hardcoded counts in prompts** (3x: #12, #23, #38 | last 1 PRs: 1 clean): Hardcodes specialist/finding counts inside agent prompts that change when fan-out changes. Examples: verifier prompt says "4 specialist reviewers" while 5 finding-blocks are attached (PR #38); verifier template hardcodes `**1. … **2. … **3.` per section (PR #23); review.md Agent N summary enumerates specific items that drift on checklist updates (PR #12). Tendency: when changing the agent fan-out or finding-block count, search every agent prompt for the literal old number — not just the verifier — and convert per-section enumerations to a templated loop.

- **Stale documentation references and metadata drift** (17x: #1, #11, #12, #14, #18, #20, #21, #22, #23, #28, #30, #33, #34, #36, #38, #39, #39 | last 0 PRs: 0 clean): Counts, descriptions, scope summaries, cost tables, version strings, badge versions, agent inventories, and architecture diagrams become stale within the same commit that changes the implementation. Highest-volume offender across the project. Recurring blast radius: `plugins/air/README.md` (mirror), `CLAUDE.md` (×2 — narrative + structure), `docs/architecture.md` (agent inventory + cost tables + roadmap + workflows tree), `managed/README.md` (cost figures + flag docs), `GLOSSARY.md` in the wiki. PR #20 left 5 mirror docs stale; PR #34 missed 4 mirror sites for "What's New"; PR #39 missed `CLAUDE.md:68`, `README.md:239`, `plugins/air/README.md:218` for the wiki-counter relocation. Tendency: before committing, grep every `.md` file under the repo for: the old version string, the old agent count, the old cost figure, the old config-file path. release-please's `extra-files` list documents the version-string mirror set; treat it as the floor, not the ceiling.

- **Self-review author mismatch** (1x: #11 | last 28 PRs: 28 clean) (archived): Used git config user.name where author.login was needed. Single occurrence, no recurrence across 28 subsequent PRs. Archived per author-pattern lifecycle.

## Compliance / Reference

These sections are reference-only and not subject to the cleanup pass.

### Severity ladder

- **blocker** — must fix before merge (security, correctness on the happy path, data loss, broken control-flow)
- **medium** — should fix before merge (incorrect on edge cases, doc drift on user-facing flags, leaky abstractions)
- **low** — fix when convenient (minor inefficiency, style inconsistency, stale comment)
- **nit** — flag for awareness; landing without a fix is fine

### Verifier verdicts

`CONFIRMED`, `DOWNGRADED`, `IMPROVEMENT`, `PRE-EXISTING`, `ACCEPTED PATTERN`, `FALSE POSITIVE`. The default confidence cutoff is 60; per-agent overrides come from SEVERITY-CALIBRATION.md when 10+ data points exist.

### Author-pattern lifecycle

`create` (first occurrence) → `strengthen` (counter increments, clean-PR counter resets to 0) → `decline` (5 consecutive clean PRs) → `archive` (10 consecutive clean PRs). Patterns are never deleted — only archived. Counter format: `(Nx: <PR refs> | last K PRs: K clean)` with `(declining)` / `(archived)` suffix when applicable.
